🐟 GuppyLM · 따라하기 (hands-on)

한·영 MoE Guppy, 직접 돌려보기

명령어를 한 줄씩 따라 치며 데이터 생성 → 토크나이저 → MoE 학습 → 라우팅 분석 → 대화까지 처음부터 끝까지 돌려봅니다. 각 단계마다 실제 출력 예시와 "이렇게 나오면 성공" 신호를 함께 둡니다.

소요: 약 15~20분 (학습 제외) · 단일 GPU 권장 · 새 의존성 없음
📘 개념·구조가 궁금하면 → moe-korean.html (왜 이렇게 바뀌는지)
0. 준비 1. 데이터 2. 들여다보기 3. 토크나이저 4. 모델 구조 5. 학습 6. 라우팅 7. 대화 8. dense 비교 9. 실습 과제 트러블슈팅

전체 흐름을 한눈에:

0 준비 1 prepare → 데이터·토크나이저 2 데이터 확인 3 토크나이저 체험 4 모델 구조 5 train → MoE 학습 6 route → 전문가 분석 7 chat 8 dense 비교 9 직접 바꿔보기

아래 출력 예시는 빠른 확인을 위해 800 샘플로 돌린 실제 결과입니다. 본 학습(기본 60,000 샘플)에서는 숫자가 더 크게 나옵니다 — 중요한 건 값이 아니라 형태와 성공 신호입니다.

소스를 GPU 머신으로 옮긴 뒤, 저장소 폴더에서 가상환경을 만들고 의존성을 설치합니다.

python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt     # torch, tokenizers — 새 의존성 없음

GPU (권장)train.py가 cuda를 자동 선택. 없으면 CPU로 느리게 동작
디스크 ~100MB데이터·토크나이저·체크포인트

1 데이터 + 토크나이저 만들기 ~1분

한 명령으로 한·영 합성 대화 생성 + BPE 토크나이저 학습이 같이 됩니다.

bash
python -m guppylm prepare

...
Training BPE tokenizer (vocab_size=8192) on 60000 texts...
Tokenizer saved to data/tokenizer.json (3160 tokens)   # 상한 8192 이내

✓ 성공 신호 출력에 By language: en=…, ko=… 가 양쪽 다 보이고, 마지막 Decoded: 줄의 한글이 안 깨지면 성공.
왜 두 언어를 50:50으로? 이중언어 모델이 한쪽으로 쏠리지 않게 하고, MoE 라우터가 두 언어 토큰을 충분히 보게 하려면 균형이 필요합니다. 한국어는 번역이 아니라 generate_data_ko.py의 네이티브 템플릿(60주제)으로 생성됩니다.

각 샘플은 text(채팅 포맷) · category(60주제 중 하나) · lang(en/ko) 세 필드를 가집니다.

왜 lang 필드? 학습 자체엔 안 쓰이지만, 데이터 언어 균형을 맞추고 나중에 "한국어가 어느 전문가로 갔는지"를 언어별로 분석할 수 있게 해줍니다(6단계).

3 토크나이저 체험 — 한글은 어떻게 쪼개지나 ~3분

한국어 문장이 토큰 몇 개로 나뉘는지 직접 봅니다. 이게 vocab을 4096→8192로 키운 이유와 직결됩니다.

bash
python - <<'PY'
from tokenizers import Tokenizer
tok = Tokenizer.from_file("data/tokenizer.json")
for s in ["안녕 구피","배고프니","밥 먹을래","hi guppy","삶의 의미가 뭐야"]:
    enc = tok.encode(s)
    print(f"{s!r:16} -> {len(enc.ids)} tokens:", [tok.decode([i]) for i in enc.ids])
PY
'안녕 구피'        -> 2 tokens: ['안녕', ' 구피']
'배고프니'         -> 3 tokens: ['배고', '프', '니']
'밥 먹을래'        -> 3 tokens: ['밥', ' 먹을', '래']
'hi guppy'     -> 2 tokens: ['hi', ' guppy']
'삶의 의미가 뭐야'    -> 6 tokens: ['�', '�', '의', ' 의미', '가', ' 뭐야']
무슨 일이 일어났나
  • 자주 나오는 단어(안녕, 구피, 밥)는 통째로 한 토큰이 됩니다.
  • 코퍼스에 드문 글자(삶)는 학습된 서브워드가 없어 raw 바이트로 쪼개져 �로 보입니다 — 한글은 UTF-8에서 1글자=3바이트라, ByteLevel BPE가 바이트 단위로 처리하기 때문입니다.
  • 그래서 한국어가 들어오면 더 많은 서브워드가 필요 → vocab 상한을 8192로 올립니다.
참고 enc.tokens를 그대로 출력하면 ìķĪëħķ처럼 보입니다. 이건 ByteLevel BPE의 내부 바이트 표현이라 정상이며, tok.decode(...)를 거치면 한글로 복원됩니다.

4 모델 구조 확인 — dense vs MoE ~2분

학습 전에, 모델이 정말 MoE로 구성됐는지와 파라미터 수를 확인합니다. 저장소에 들어있는 check_model.py를 저장소 루트에서 실행하세요.

bash
python check_model.py
GuppyLM (dense): 10,304,... params (10.3M)
GuppyLM (MoE 4x top-2): 20.9M total / 13.9M active per token

스크립트 내용은 다음과 같습니다 — 직접 만들어 보려면 이 코드를 check_model.py로 저장하세요.

check_model.py
from guppylm.config import GuppyConfig
from guppylm.model import GuppyLM

for moe in (False, True):
    m = GuppyLM(GuppyConfig(use_moe=moe))
    print(m.param_summary())
전체(메모리/용량)토큰당 활성(연산)
dense10.3M10.3M
MoE 4·top-220.9M13.9M
✓ 성공 신호 MoE 줄에 20.9M total / 13.9M active가 보이면 모델이 제대로 전문가 4개로 구성된 것입니다. 용량은 ~2배인데 토큰당 연산은 절반 수준만 늘었다는 게 MoE의 핵심입니다.
자주 나는 에러 두 가지
  • ModuleNotFoundError: No module named 'guppylm' — 저장소 루트가 아닌 곳에서 실행한 경우입니다. guppylm은 PyPI 패키지가 아니라 저장소 안의 로컬 폴더라서, guppylm/ 폴더가 보이는 위치로 cd 한 뒤 다시 실행해야 합니다.
  • IndentationError: unexpected indent — 코드를 대화형 파이썬 셸(>>>)에 한 줄씩 붙여넣을 때 앞에 공백이 끼어 발생합니다. REPL에 붙여넣지 말고 위처럼 python check_model.py 파일로 실행하세요.
python: command not found가 나오면 python3 check_model.py로 실행하세요.

이 프로젝트의 핵심 질문: 전문가들이 분화했는가? 한국어와 영어가 다른 전문가로 가는가?

읽는 법 위처럼 expert 0이 한국어, expert 1이 영어로 쏠리면 언어 분화가 일어난 것입니다. 꼭 이렇게 깔끔하게 안 갈려도 정상입니다 — 실제 MoE는 의미보다 조사·구두점 같은 표면 패턴으로 갈리는 경우가 많습니다. 중요한 건 분포가 한 전문가로 100% 붕괴하지 않는 것.
충분히 학습 안 한 모델은? 스텝이 적으면 네 전문가가 모두 (shared)로 비슷하게 나옵니다 (분화가 아직 안 생김). 이것도 "붕괴는 아니다"라는 점에서 정상 — 더 오래 학습하면 갈라질 수 있습니다.

9 직접 바꿔보기 실습 과제

쉬움 전문가 수를 8개로

config.py에서 n_experts=8로. 다시 train → route.

예상: 전체 파라미터가 더 커지고, 전문가당 데이터가 줄어 분포가 더 흩어집니다. param_summary로 전체/활성 변화 확인.

쉬움 top-1으로 (Switch 스타일)

moe_top_k=1로. 토큰당 전문가 1개만 사용.

예상: 활성 파라미터가 더 줄고(더 빠름), 라우팅이 불안정해지기 쉬움(Aux가 더 출렁). 학습 안정성을 top-2와 비교.

중간 부하분산 세기 조절

aux_loss_coef를 0.01 → 0.1 또는 0.001로.

예상: 크게 하면 전문가를 더 강하게 균등화(분화 약해짐), 작게 하면 분화가 세지지만 붕괴 위험↑. route의 lean 변화를 관찰.

중간 새 한국어 주제 추가

generate_data_ko.py에 _topic_ko(user_msgs, guppy, "새주제")로 생성기를 만들고 KO_GENERATORS에 추가. (영어도 짝을 맞추려면 generate_data.py에도 동일 카테고리 추가)

예상: prepare 출력의 카테고리 목록에 새 주제가 등장. Guppy 말투(짧은 반말, 물고기 시점)를 유지하는 게 포인트.
